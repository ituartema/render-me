How access works in the enterprise app

A shared model for how roles, activities and scope compose — built from real field cases and engineering conversations. Some parts are settled; some are open questions still being worked. It's a model to think with, not a final spec.

Where it stands

Common ground

— The role is the entry point to the experience; a config defines that experience.
— Reusable, hierarchical configs (define once, reuse) are on the table.
— Edits are events (engineering-confirmed): an edit appends an immutable event; the projection it also updates is a rebuildable cache.
— Role-based access is the right frame; usage access can follow from user access.

Still open

— Assignment: how a config attaches to a user — a permission or a semantic link.
— Form dropdowns list all accessible entities regardless of the view selector — needs form selectors extended.
— Region doesn't exist yet; grouping states into regions is gated on list-of-entities (being built).
— A likely third (partner) hierarchy; and whether a Head of Section sees head teachers.

Two open deltas this surfaced

1 · "Activity" vs "plugin" — likely the same thing

Product's activity (a reusable bundle for a job) and engineering's plugin (a configurable feature component) appear to be the same construct named from two sides. Treated here as one thing, pending final confirmation — a vocabulary convergence, not a conflict.

2 · Does scope need to be its own primitive?

This model treats scope as its own primitive, because today it's reconstructed fragilely from roles, coverage and links. An alternative view holds scope needs no separate primitive — role-based access rights inside each config could express the same thing. Both are on the table. The cleanest way to settle it: build a real config for the hardest case — the scope lattice — and see whether it stays clean.
