The entity map, and how access paths decide what a user sees

A Trellis organization is a database you design from scratch: entity types are the tables, references are the wiring between them, and access-control paths walk that wiring to scope exactly what each role can see. Work through it top to bottom, or jump straight to the live map and the path tracer.

02 — the building blocks

Entity kinds

Every entity type has a kind — the behaviour class that tells the system how it should act. Four kinds appear in this org; most things are Generic, and a few earn special powers.

Generic

Just data

A name and some fields, no special behaviour. Most of the org: geography, roles, lookups, classes, learners.

School

Issues school accounts

The anchor entity. Everything downstream — teachers, classes, dashboards — feeds off it.

User

Gets a login

A Keycloak account. EIDU Staff, Government Officials, and — newly — Teachers, who are app users without a Console role.

Form

Carries forms

Holds form definitions and availability. One of the data-oriented kinds the platform grew into.

References & labels

A reference is a field pointing at another entity — one direction, one hop. A label is what shows in a list: the name, plus an optional second line pulled from a reference to tell near-identical records apart. Click any node in the map above to see both.

The machinery: config, runtime, feature

A config is the packaged view a role gets. A runtime renders an asset into an entity that syncs to the field app; a feature is the same in raw JS but leaves no asset. Engine vs. car — and runtime & feature are transitional, being baked into the core as part of MLP.

03 — teaching the chain

Semantic links

References stop after one hop, so on its own the system can't follow a chain across several entities. A semantic link declares the whole route so analytics can walk it end to end. Toggle the two views.

Set onceFirst-time configuration, after entity types exist. Stable — you rarely touch it.
No editsLinks can't be edited, only deleted and recreated.
Separate todayEncodes the same relationships as access control; there's a plan to unify them, not yet prioritised.
UnownedNobody actively maintains this — a natural job for the forward deployment team.
04 — the hard part

Access-path tracer

References only reach one level deep, so the Console can't follow a chain on its own. To scope what a role sees, you declare a path from the user out to the target entity. Pick a role and what they need to see — the path lights up.

05 — what reaches the field app

Configs & the field app

A config is the role-specific view someone opens in the field app. It's a stack: data sets feed charts, charts fill dashboards, and dashboards make up the config.

The query underneath — where the numbers come from.

Which config a person sees is wired through access control, matched to their role. Change the role, and the view changes with it — powerful, and a reason role edits need care.

Built in code today

Engineers create configs for now; AI-assisted creation by product is the intended direction. Runtimes and features are the machinery rendering it underneath.

Impersonate stops short

Admin impersonate previews a user's entities, but not their configs or dashboards yet — so you can't fully preview a head teacher's view. A good hackathon target.

06 — how to run it

Operating principles

Not features — judgement calls earned from a year of running IDO 2.0. Challenge them; they're assumptions, not laws.

Fewer hands, more accountability

One owner, maybe a backup. Many teams writing occasionally and under pressure produces expensive, avoidable mistakes.

Collaboration has a price

Multi-team editing would first need real validation, notifications, and approval steps. Functionality came before design polish — you can't have both at once.

No prod / test split

Retired as too costly. Spin up a throwaway org or use the demo org — neither gives production-scale data, which is the standing gap.
